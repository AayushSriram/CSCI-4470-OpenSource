Testing

  • Testing of Systems
  • Testing of Code

Purpose

  • Software Quality Control/Software Quality Assurance
  • Validation - building the right product
  • Verification - does the code/system implement the specification
  • Possibly avoid debugging

Debugging vs Testing

  • Debugging is a cyclic activity - code execution and correction
  • Aim of Debugging is to locate and correct errors
  • Testing is to demonstrate "correctness"

Testing of Code

  • Black Box - Treat the system as black box and try test cases based on input/output specification
    • Often used for "Acceptance" or "User" testing
  • Gray Box - Has limited knowledge of the system and try test cases based on it. Correctly cleaning up itself
    • Often used for <"Integration" or "System" testing
  • White Box - Code details are known. Tests are based on the code.
    • Often used for "Unit" testing

Types of Tests

  • Smoke Tests - most common kind of test - non extensive - most crucial functions of a program work
    • "Acceptance" or "User" testing / Black Box
  • Functional Tests - Done by QA to test functionality according to a test plan based on requirements and design specs.
    • "Integration" or "System" testing / Gray Box
  • Unit Tests - Done by developers to test specific code.
    • Well, "Unit" testing / White Box
  • Regressions Tests - Change in code does not affect other parts
    • Any or all ... Often automated

Why Write Unit Tests

  • Increase Developers Confidence
  • Avoid regression - If unit test is run frequently, one knows when new code breaks old code.
  • if you write the test first, you know when you are done.
  • Encourage maximal modularity and minimal interface

When to Write Unit Tests

  • Always
  • Before you check the code into repository to know and demonstrate your code works.
  • Before and after refactoring - redesign/reimplementation does not break the code
  • Before Debugging, to ease the process and let you know when you have finished debugging.

Foundational Quote

If it ain't tested, it's broken! -- Bill Lorensen, GE Global Research Center

Terminology

  • An error is made by an Engineer/Algorithm Designer/Implementor
  • A fault is manifestation of that error in the code
  • A failure is an incorrect output behavior that is caused by executing a fault
  • Testing attempts to discover failures in software systems
    • Debugging associates failures with faults and corrects them.
  • If a system passes all of its tests, is the system free of faults? - No

Why No

  • Faults may be hiding in portions of code that rarely get executed and may not be tested.
  • Sometimes faults mask each other
  • Sometimes faults are only exhibited with specific inputs or sequences
  • It is not possible to create a test that covers all code paths and all functional units
  • Having all tests pass increases our confidence that our system has high quality

Looking for Faults

  • The input/output space of any software system is vast
  • Tests are a way of sampling the behaviors of a software system looking for failures
  • Partition the space into equivalent behaviors and sampling each partition

Example

  • The GCD program takes two numbers and computes the GCD 

def gcd(a,b):
    while(1 > 0):
        if (a == 0):
            return b
        b = b % a
        if (b == 0):
            return a
        a = a % b

Test Cases for the GCD Program

  • Assumptions in data a and b are nonnegative integers. Try edge case
  • Assumptions about Data are implemented in the program.
  • a = 9, b =3 gcd = 3
  • a = 2, b = 6 gcd = 2
  • a = 3 , b= 11 gcd =1 prime numbers
  • a = 0, b = 10 gcd=? edge case
  • a = -9, b =18 gcd=? data constraint

Test Coverage

  • Covers all the paths
  • Covers all the loops and conditionals
  • Statement Coverage - all statements have been executed.
  • Branch Coverage - all edges in the control graph have been executed at least once .
  • Condition Coverage - all combination of conditions have been covered.
    • Or have they:
    • a = 18, b =-9 gcd=?

Unnecessary_math.py
'''
Module showing how doctests can be included with source code
Each '>>>' line is run as if in a python shell, and counts as a test.
The next line, if not '>>>' is the expected output of the previous line.
If anything doesn't match exactly (including trailing spaces), the test fails.
'''
 
def multiply(a, b):
    """
    >>> multiply(4, 3)
    12
    >>> multiply('a', 3)
    'aaa'
    """
    return a * b

Unittest

 test_um_unittest.py 
import unittest
from unnecessary_math import multiply
 
class TestUM(unittest.TestCase):
 
    def setUp(self):
        pass
 
    def test_numbers_3_4(self):
        self.assertEqual( multiply(3,4), 12)
 
    def test_strings_a_3(self):
        self.assertEqual( multiply('a',3), 'aaa')
 
if __name__ == '__main__':
    unittest.main()
